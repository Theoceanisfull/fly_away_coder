# Experiment 01 Results

## Title

GEPA prompt-optimization sweep over local Ollama coding models on SciCode and LiveCodeBench, with comparison to published local baseline benchmark snapshots.

## Executive Summary

The main overnight experiment was a completed four-run GEPA sweep executed from **2026-04-20 01:17:58 UTC** to **2026-04-20 10:21:19 UTC** (**2026-04-19 09:17:58 PM EDT** to **2026-04-20 06:21:19 AM EDT**). The best result was on **LiveCodeBench with `rnj` as the student model**, where mean score improved from **0.1373** to **0.1601** and solved problems increased from **22** to **26** on the held-out evaluation split. A smaller gain was observed on **SciCode with `devstral` as the student model**, where mean score improved from **0.2305** to **0.2429**, but solved-count did not change.

Two runs showed no accuracy gain:

- `scicode` + `rnj`: **0.1640 -> 0.1640**
- `livecodebench` + `devstral`: **0.2011 -> 0.2011**

Relative to the repo's published full-benchmark baseline snapshots, the overnight GEPA sweep was:

- **directionally better on SciCode**, when compared against the published SciCode subproblem-level correctness numbers
- **still materially below the published LiveCodeBench baseline**, for both models

These comparisons are useful but **not directly apples-to-apples**, because the overnight GEPA sweep was evaluated on held-out split subsets inside the prompt-optimization harness, whereas the published baselines were generated by separate full benchmark harnesses over much larger evaluation sets.

## Methods

### Models

The experiments used the local Ollama-served coding models configured in `experiments/prompt_optimization/run_gepa.py`:

- `rnj` -> `rnj-1:latest`
- `devstral` -> `devstral-small-2:latest`

The student LM and reflection LM were both served from:

- `api_base = http://localhost:11434`

The default cross-reflection pairing was:

- `rnj-1:latest` student -> `devstral` reflection LM
- `devstral-small-2:latest` student -> `rnj` reflection LM

### Prompt-Optimization Procedure

The optimization harness uses DSPy GEPA with a simple code-generation module:

- `CodeSolver = dspy.ChainOfThought(SolveCodeTask)`
- input fields: benchmark task prompt and starter code
- output field: executable Python 3 code only

For each run, the workflow was:

1. Load benchmark tasks.
2. Build disjoint train, validation, and evaluation splits.
3. Evaluate the unoptimized student program on the held-out evaluation split.
4. Run `dspy.GEPA(...).compile(...)` using the train and validation splits.
5. Save the optimized program.
6. Re-evaluate the optimized program on the same held-out evaluation split.

Each run used the following core settings:

- `temperature = 0.0`
- `max_tokens = 2048`
- `timeout_s = 12.0`
- `num_threads = 1`
- `max_full_evals = 2`
- `seed = 0`

### Benchmarks and Split Construction

#### SciCode

SciCode tasks were loaded from:

- Hugging Face dataset `SciCode1/SciCode`
- split: `test`

Hidden targets were loaded locally from:

- `benchmarks/scicode/data/test_data.h5`

The prompt-optimization harness evaluates **SciCode sub-steps**, not full end-to-end main problems. Each SciCode evaluation score is the fraction of hidden tests passed for that sub-step.

For the full overnight sweep, SciCode used group-count splits by `problem_id`:

- `train_groups = 24`
- `val_groups = 8`
- `eval_groups = 16`

This produced:

- `train_examples = 96`
- `val_examples = 43`
- `eval_examples = 94`

#### LiveCodeBench

LiveCodeBench tasks were loaded from the local dataset snapshot:

- `benchmarks/livecodebench/code_generation_lite/`
- release tag: `release_latest`

The harness decodes hidden tests from the dataset blob and supports both:

- standard input tasks
- functional or LeetCode-style tasks

LiveCodeBench groups were defined by:

- `(contest_date, contest_id)`

This preserves a temporally held-out evaluation split. For the full overnight sweep, LiveCodeBench used:

- `train_groups = 40`
- `val_groups = 10`
- `eval_groups = 40`

This produced:

- `train_examples = 159`
- `val_examples = 44`
- `eval_examples = 175`

### Evaluation Metric

For both benchmarks, the reported per-example score is:

- `score = passed_hidden_tests / total_hidden_tests`

The main aggregate reported by the GEPA harness is:

- mean score across all held-out evaluation examples

The harness also reports:

- solved count: number of evaluation examples with perfect hidden-test completion
- partial count: number of evaluation examples with non-zero but non-perfect score
- zero-score count
- average time per evaluation example
- improved, regressed, and unchanged row counts after optimization

The feedback exposed to GEPA is deliberately coarse and typically reports only:

- `wrong_answer`
- `timeout`
- `runtime_error`

with the first failing hidden test index.

### Baseline Comparison Method

The overnight GEPA results were compared against the repo's published benchmark snapshots:

- SciCode baseline snapshot generated at `2026-04-03T11:46:59+00:00`
- LiveCodeBench baseline snapshot generated at `2026-04-08T03:44:15+00:00`

These published baseline snapshots represent **full benchmark harness runs**, not the smaller GEPA optimization splits. Therefore:

- SciCode comparisons are **directional** comparisons between published subproblem correctness and held-out GEPA mean score
- LiveCodeBench comparisons are **directional** comparisons between published `pass@1` or full-benchmark solved counts and held-out GEPA mean score or solved counts

The results should therefore be treated as **within-harness optimization evidence**, not as final benchmark replacement numbers.

### Main Observations

1. **The clearest accuracy improvement occurred on LiveCodeBench with `rnj`**.
   The optimized prompt increased mean score by **0.0229** and increased solved problems from **22** to **26** on the held-out split.

2. **SciCode with `devstral` improved modestly but did not solve more sub-steps perfectly**.
   The gain came from more partial-credit improvements rather than more fully solved examples.

3. **Two runs produced no measurable accuracy gain**.
   This occurred for `scicode/rnj` and `livecodebench/devstral`.

4. **Latency changed substantially in some cases even when accuracy did not**.
   The most extreme case was `livecodebench/devstral`, where average per-example time dropped from **26.84 s** to **0.74 s** with no change in score. `scicode/rnj` also dropped from **7.61 s** to **1.30 s** without changing accuracy.

5. **The sweep is best interpreted as mixed but non-trivial evidence that GEPA can help the weaker student on LiveCodeBench**, while benefits on SciCode were smaller and less consistent.

## Analysis

### What Worked

The strongest result was the `rnj` student on LiveCodeBench. This run improved both:

- average held-out score
- number of fully solved held-out problems

That pattern is stronger evidence than a mean-score gain alone because it suggests actual end-task success increased, not merely partial hidden-test improvement.

The SciCode `devstral` run also improved, although the gain was smaller and took the form of increased partial credit rather than more perfect solves. This suggests that GEPA may have nudged the prompt toward somewhat better scientific code behavior without crossing enough thresholds to create more fully passing sub-steps.

### What Did Not Work

The `scicode/rnj` and `livecodebench/devstral` runs showed zero change in accuracy. In both cases, the optimizer appears to have converged to prompts that preserved evaluation behavior rather than improving it.

The standalone one-example probe runs were not informative. They all failed on the same SciCode hidden tests and therefore offered no evidence of prompt improvement. These runs should be treated as debugging artifacts rather than substantive experiments.

### Why SciCode and LiveCodeBench Behaved Differently

A plausible interpretation is that the prompt space for LiveCodeBench and SciCode differs materially:

- LiveCodeBench includes many standard competitive-programming and LeetCode-style tasks where prompt framing may meaningfully affect code structure and correctness
- SciCode sub-steps often require more specific scientific formulas, numerical details, or domain reasoning, so prompt optimization alone may have less leverage when the base model lacks the required scientific competence

This is consistent with the observed pattern:

- `rnj` gained on LiveCodeBench
- `devstral` gained slightly on SciCode
- neither benchmark showed broad, uniform improvement across both models

### Interpreting the Large Latency Drops

Two zero-delta runs showed very large reductions in average per-example generation time. The most likely explanation is that GEPA sometimes found prompts that induced much shorter model outputs without changing final hidden-test behavior. This may still be practically useful if the resulting programs remain equally correct, but it should not be interpreted as an accuracy gain.

### Threats to Validity

1. **The overnight GEPA runs were not full benchmark reruns.**
   They used held-out evaluation splits inside the prompt-optimization harness, not the published full-benchmark harnesses.

2. **SciCode metrics are sub-step oriented.**
   The overnight SciCode runs optimize and score sub-steps rather than full end-to-end problems.

3. **The sample sizes were smaller than the published baselines.**
   The full published baselines use 65 SciCode problems and 1055 LiveCodeBench problems, while the overnight sweep used 94 SciCode evaluation sub-steps and 175 LiveCodeBench evaluation tasks.

4. **The optimization budget was small.**
   Each run used `max_full_evals = 2`, which is intentionally lightweight and may underexplore the prompt space.

5. **Reflection-model effects are entangled with prompt effects.**
   The cross-reflection setup may have helped some student models more than others.

## Conclusions

The completed overnight GEPA sweep provides evidence that prompt optimization can improve held-out coding performance in this setup, but the effect is selective rather than universal.

The main findings are:

1. **LiveCodeBench with `rnj` showed the most meaningful gain**.
2. **SciCode with `devstral` showed a modest gain**.
3. **The other two full sweep runs did not improve accuracy**.
4. **The overnight GEPA results do not yet surpass the repo's published LiveCodeBench baseline snapshots**.
5. **On SciCode, the overnight results are directionally better than the published subproblem-level baselines, but still require full-harness confirmation**.

## Recommended Next Steps

1. Re-run the full benchmark harness with the optimized prompts from the two promising runs:
   - `livecodebench/rnj`
   - `scicode/devstral`

2. Export those full-harness outputs into new timestamped baseline snapshot folders for true benchmark-to-benchmark comparison.

3. Increase the GEPA search budget beyond `max_full_evals = 2` to test whether the current gains were budget-limited.

4. Inspect the per-example comparison CSV files for the improved runs to determine whether gains cluster by task family, difficulty, or contest era.

5. Clean up or explicitly mark stale `status.json` files for interrupted probe sweeps to avoid ambiguity in later reporting.
